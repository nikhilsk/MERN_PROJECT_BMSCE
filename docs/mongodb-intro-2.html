‚ùìWhy do we even compare SQL and MongoDB?

Think like an app developer, not a database engineer.

SQL ‚Äì Like strict Excel sheets
Great when rules never change.
  • Tables, rows, columns.
  • Perfect for banks, payroll, marks database.
  • Very strong rules (schema, constraints).
MongoDB ‚Äì Like flexible Google Docs
Great when your app keeps evolving.
  • Stores JSON‚Äëlike documents.
  • Easy to store nested data (orders with items).
  • Add new fields any time, old data still works.

Rule of thumb for students:

  • If your UI screens are naturally JSON‚Äëshaped (like an order bill, user profile), MongoDB will usually feel simpler.
  • If your world is reports and strict rules (banking, accounting), SQL will usually fit better.

üßæOne Screen Story: The Bill

We only design one screen: the order bill in a food app.

Customer: Alice Items: Pizza √ó2, Salad √ó1 We need: name + items + total

1. SQL version ‚Äì split across 3 tables

-- Customers
+----+----------+
| id |  name    |
+----+----------+

+----+----------+

-- Orders
+----+------------+-----------+

+----+------------+-----------+
| 1  |    10      | 2025-12-08|
+----+------------+-----------+

-- OrderItems
+---------+----------+-----+-------+
| orderId | itemName | qty | price |

-- To draw ONE bill:
SELECT c.name, oi.itemName, oi.qty, oi.price
FROM Customers c
JOIN Orders o ON c.id = o.customerId
JOIN OrderItems oi ON o.id = oi.orderId
WHERE o.id = 1;

For just one screen, beginners must learn:

  • How foreign keys work (customerId, orderId).
  • How JOINs work (Customers ‚Üî Orders ‚Üî OrderItems).

2. MongoDB version ‚Äì one JSON document

Now imagine we just save the bill exactly how the UI needs it.

// One document in the "orders" collection
{
  _id: 1,
  customer: {
    id: 10,
    name: "Alice"
  },
  orderDate: "2025-12-08",
  items: [
    { name: "Pizza",  qty: 2, price: 8.99 },
    { name: "Salad",  qty: 1, price: 5.99 }
  ],
  total: 23.97
}

To load the bill screen:

db.orders.findOne({ _id: 1 })

Done. One collection, one document, one query.

Analogy:

  • SQL: Marks in 3 different notebooks (Math, Science, English). To show report, teacher opens all and combines.
  • MongoDB: One report card that already has all subjects written together.

üß∞Atlas, Compass & Mongoose

These three tools make MongoDB very friendly for MERN learners.

MongoDB Atlas
Cloud MongoDB (no install)
  • Create a free cluster.
  • Connect from anywhere (college, home).
  • Great for demos and projects.
MongoDB Compass
Visual GUI for your data
  • See collections and documents like folders and files.
  • Run queries without writing code.
  • Perfect for understanding how embedding looks.
Mongoose
Bridge between Node.js and MongoDB
  • Defines schemas in JavaScript.
  • Adds validation (required fields, types).
  • Makes queries feel like normal JS methods.

In a MERN app:

  • Frontend (React) sends JSON ‚Üí Backend (Express) receives JSON ‚Üí MongoDB stores JSON‚Äëlike documents.
  • No ‚Äútranslation‚Äù headache between objects and tables.

End‚Äëto‚ÄëEnd Flow: From React to MongoDB

Let‚Äôs follow one new order from the browser all the way to the database, step by step.

React form ‚Üí shape the JSON Express API ‚Üí calculate total Mongoose model ‚Üí validate & save MongoDB ‚Üí embedded document

1) React: build the order object

// React (frontend)
import axios from 'axios';

function placeOrder() {
  const order = {
    customer: { id: 10, name: 'Alice' },
    items: [
      { name: 'Pizza', qty: 2, price: 8.99 },
      { name: 'Salad', qty: 1, price: 5.99 }
    ]
    // Notice: we are NOT sending total yet.
  };

  axios.post('http://localhost:3001/orders', order);
}

Important: The React object is already in the embedded structure we want:

  • customer is an object inside the order.
  • items is an array of objects inside the order.

2) Express: calculate the bill total

// Express (backend)
app.post('/orders', async (req, res) => {
  // 1. Read the JSON sent by React
  const data = req.body;  // { customer: {...}, items: [...] }

  // 2. Calculate the total on the server
  const total = data.items.reduce((sum, item) => {
    return sum + item.qty * item.price;
  }, 0);

  // 3. Create the final object to save (now includes total)
  const orderToSave = {
    customer: data.customer,
    items: data.items,
    total: total
  };

  // 4. Save using Mongoose (this will embed items inside the order document)
  const newOrder = await Order.create(orderToSave);
  res.json(newOrder);
});

Where did embedding happen?

  • We passed one big object to Order.create(orderToSave).
  • This object has customer (object) and items (array) inside it.
  • Mongoose + MongoDB store that as one document with embedded fields.

3) Mongoose: schema that matches our JSON shape

// Mongoose model (Node.js)
const mongoose = require('mongoose');

const OrderSchema = new mongoose.Schema({
  customer: {
    id: Number,
    name: String
  },
  orderDate: { type: Date, default: Date.now },
  items: [{
    name: String,
    qty: Number,
    price: Number
  }],
  total: Number
});

const Order = mongoose.model('Order', OrderSchema);

Visual: This schema says:

  • "Inside each order document, there is a customer object".
  • "Inside each order document, there is an array items, and each item has name, qty, price".

4) MongoDB: final embedded document in the collection

// In the "orders" collection in MongoDB
{
  _id: ObjectId("..."),
  customer: { id: 10, name: "Alice" },
  orderDate: ISODate("2025-12-08T10:00:00Z"),
  items: [
    { name: "Pizza",  qty: 2, price: 8.99 },
    { name: "Salad",  qty: 1, price: 5.99 }
  ],
  total: 23.97
}

Notice the journey:

  • React order object ‚Üí sent as JSON.
  • Express adds total and creates orderToSave.
  • Mongoose schema describes the embedded structure.
  • MongoDB stores exactly that structure in one embedded document.
